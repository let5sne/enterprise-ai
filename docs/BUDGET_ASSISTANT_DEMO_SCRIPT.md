# BUDGET_ASSISTANT_DEMO_SCRIPT

## 1. 3 分钟演示话术

适合场景：

- 领导临时看一下
- 内部快速汇报
- 客户首次接触时的短演示

推荐讲法：

“这个系统不是做一个通用聊天机器人，而是把预算分析这件事做成一条完整链路。用户先问一个预算问题，系统先做数据分析，再返回表格、图表和可读结论；如果继续追问，系统会保留上下文继续展开；最后还可以把分析结果直接整理成给领导看的经营说明。”

建议现场演示顺序：

1. 输入：`本月哪些部门超预算？`
2. 强调返回了 `answer + table + chart + raw_sql`
3. 输入：`按预算科目展开一下`
4. 强调它不是重新问，而是在延续上一轮分析
5. 输入：`写成给领导看的经营分析说明`
6. 强调它已经从分析走到了表达

收尾一句话：

“这说明它已经不是单点能力，而是一个可演示、可试点的预算助手闭环。”

## 2. 10 分钟演示话术

适合场景：

- 正式内部评审
- 客户售前交流
- 合作方能力介绍

推荐结构：

### 第一段：先讲问题，不先讲技术

“企业预算分析常见的问题不是没有数据，而是数据散、响应慢、表达难。领导问一个问题，往往需要业务、财务、经营分析几个人来回拉数、对口径、写说明。”

### 第二段：再讲系统定位

“我们现在做的不是替代 ERP，也不是替代 BI，而是在问问题、拉分析、继续追问、生成汇报说明这一条链路上，提供一个更轻、更快、更适合试点落地的能力层。”

### 第三段：进入单轮演示

演示：

```json
{"user_id":"u1","source":"web","message":"本月哪些部门超预算？"}
```

讲解重点：

- 系统识别为 `data.analyze`
- 输出的不是一句话，而是结构化 `artifacts`
- 可以继续被前端、IM、报告系统消费

### 第四段：进入 follow-up 演示

演示：

```json
{"session_id":"sess_budget_demo","user_id":"u1","source":"web","message":"按预算科目展开一下"}
```

讲解重点：

- 系统识别为 `data_followup`
- 说明它能记住上下文，而不是每次都当成新问题
- 这对经营分析场景很关键，因为真实业务里领导的提问往往是连续追问

### 第五段：进入经营说明演示

演示：

```json
{"session_id":"sess_budget_demo","user_id":"u1","source":"web","message":"写成给领导看的经营分析说明"}
```

讲解重点：

- 系统切换到 `content.generate`
- 利用上一轮结构化分析结果生成正式说明
- 这一步体现的是“从分析到表达”的衔接能力

### 第六段：最后落到试点价值

“所以这个阶段最适合做的不是继续堆更多功能，而是用它去做试点验证，证明预算分析和经营说明生成这条链路到底能不能显著提高业务响应效率。”

## 3. 领导版讲法

领导通常不关心实现细节，更关心三件事：

- 能不能更快回答问题
- 能不能减少人工整理
- 能不能进入日常汇报

推荐讲法：

“这个系统的价值不是替代现有系统，而是在领导提问以后，把原来需要多人手工完成的分析和说明整理，压缩成一次连续交互。现在它已经能完成预算分析、连续追问和汇报说明生成，下一步就适合在一个试点场景里接真实数据做验证。”

领导版不要重点讲：

- 模型细节
- 框架细节
- Prompt 细节
- 代码结构

领导版重点讲：

- 响应效率
- 管理口径统一
- 试点投入小
- 后续可接真实系统

---

## 4. 客户版讲法

客户更关心的是：

- 对他们当前痛点有没有针对性
- 会不会把项目讲得太大
- 能不能快速开始试点

推荐讲法：

“我们现在给您看的不是一个大而全平台，而是一个预算助手试点样板。它已经能处理部门超预算、项目超预算、月度对比、预算科目展开，以及基于分析结果自动生成经营说明。这样您可以先在一个小范围试点，验证价值，再决定是否接入真实系统和继续扩展。”

客户版重点强调：

- 试点范围小
- 可以先接只读数据
- 不要求先改造客户现有系统
- 先证明价值，再谈扩展

不要对客户直接承诺：

- 已经完成真实 ERP 深度集成
- 已经具备完整权限体系
- 所有预算问题都能覆盖

## 5. 常见质疑与回答

### 质疑 1：这是不是在替代 BI？

回答：

“不是。BI 更像固定报表和分析平台，这个助手更适合处理高频问答、连续追问和经营说明生成，它是补充层，不是替代层。”

### 质疑 2：是不是必须先接 ERP 才能用？

回答：

“不需要。当前最合适的是先做试点，把固定问题集和真实数据口径接起来，先验证价值，再逐步接真实系统。”

### 质疑 3：如果领导继续追问怎么办？

回答：

“当前系统已经支持多轮 follow-up，可以围绕上一轮分析继续展开，比如从部门超预算继续按预算科目拆开，再进一步生成正式说明。”

### 质疑 4：是不是只能做 Demo，不能落地？

回答：

“当前阶段确实是试点样板，但它已经不是静态 PPT Demo，而是有统一接口、结构化输出、上下文能力和 smoke 演示脚本的可运行系统，下一步就是接真实数据做业务试点。”

### 质疑 5：为什么不一次做成大平台？

回答：

“预算分析这类场景如果一开始就做成大平台，项目风险很高。现在更合理的路线是先做最小闭环，先验证价值，再决定扩展范围。”

## 6. 演示失败时的兜底话术

现场演示最常见的失败不是代码报错，而是：

- 网络或环境问题
- 讲解节奏失控
- 对方把问题带到系统边界之外

推荐兜底方式：

### 情况 1：环境波动

话术：

“现场环境有波动，我先不纠结操作细节，直接讲清楚这条链路的核心价值：预算问题可以从分析、展开到汇报说明形成闭环。演示脚本和测试结果我们都有留档，后面可以单独给您完整跑一遍。”

### 情况 2：对方追问超出边界

话术：

“这个问题已经进入真实系统接入和交付边界了。当前这版不是承诺所有能力都已完成，而是证明预算助手试点样板已经成立，下一步就可以进入真实数据接入和试点范围设计。”

### 情况 3：对方质疑 Demo 不够大

话术：

“我们刻意没有把它讲成一个大而全平台，因为那样很容易失焦。当前这版的价值是：小范围、低风险、可快速试点，先证明价值，再决定扩展投入。”

### 情况 4：需要快速收尾

话术：

“我用一句话总结：这个系统已经把预算分析从单点查询推进成了可连续追问、可自动生成说明的试点闭环，适合下一步直接进入真实业务验证。”
