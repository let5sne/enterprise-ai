# BUDGET_ASSISTANT_PRODUCT_BRIEF

## 1. 解决什么企业问题

企业在预算执行和经营分析上，通常有三个共性问题：

- 数据散：预算、实际发生、项目、部门、科目分散在不同表和不同系统里，汇总成本高。
- 响应慢：领导或业务部门提出问题后，财务和经营分析岗位往往需要手工拉数、整理口径、再写说明。
- 表达难：即使拿到了数据，也很难快速把结果转成面向管理层的简洁经营结论。

预算助手的目标，不是替代现有 ERP 或 BI，而是在“问问题 -> 拉分析 -> 继续追问 -> 形成汇报说明”这条链路上，提供一个更快、更轻、更适合试点落地的能力层。

---

## 2. 当前 Demo 能做什么

当前 Demo 已经具备一个稳定、可演示的最小闭环，能够通过统一接口完成预算相关问答。

当前已支持的典型问题：

1. 本月哪些部门超预算
2. 哪个项目超预算最多
3. 本月成本和上月相比变化如何
4. 按预算科目继续展开分析
5. 基于分析结果生成给领导看的经营分析说明

系统输出不只是自然语言答案，还包括：

- 结构化 `table` artifact
- 可视化 `chart` artifact
- 可追溯的 `debug.raw_sql`
- 可继续追问的轻量 follow-up context
- 面向管理层的 `text` artifact

这意味着它既能“回答”，也能“被前端、报表、IM、文档系统继续消费”。

---

## 3. 系统能力架构

从能力视角看，这个系统可以拆成三层：

### 统一入口层

- 统一接口：`POST /api/v1/chat/ask`
- 统一响应协议：`answer / artifacts / debug / task_context`

### 编排层

- 识别当前请求属于数据分析还是内容生成
- 识别是否是 follow-up
- 继承上一轮轻量上下文
- 把分析和表达串成一条可追溯链路

### 能力层

- `data.analyze`：预算 / 业财问题分析
- `content.generate`：经营说明生成
- rich artifacts：输出表格、图表、文本

这套结构的价值在于：后续可以继续接真实系统，但外部调用方式和前端消费方式不需要推翻重来。

## 4. 适合哪些客户场景

当前阶段最适合以下场景：

- 预算管理试点：预算执行分析、超预算追踪、经营月报辅助
- 财务共享 / 经营分析部门：把高频问答和说明生成从手工处理变成半自动
- 项目制企业：按部门、项目、预算科目快速穿透分析
- 售前 PoC：向客户展示“AI 如何把数据分析和管理表达串起来”

不适合当前直接承诺的场景：

- 全量替代 BI 平台
- 复杂多租户 SaaS
- 真实 ERP 深度集成已完成
- 大规模权限体系和审计体系已完整上线

正确定位应该是：这是一个适合试点验证价值的预算助手能力样板，而不是一个已经完工的大平台。

## 5. 下一步如何接真实系统

从试点走向真实交付，推荐按三步推进：

### 第一步：替换 Demo 数据

把当前内存 Demo 数据替换成客户真实数据源，只读接入即可。优先接：

- 预算台账
- 实际发生台账
- 部门维表
- 项目维表
- 预算科目维表

### 第二步：统一口径

先把最核心的管理口径固定下来，例如：

- 超预算金额
- 超预算率
- 月度对比口径
- 预算主体维度

这个阶段的关键不是功能多少，而是让“同一个问题每次都返回同一种解释”。

### 第三步：接业务流程

当分析口径稳定后，再逐步接：

- 登录 / 组织权限
- 真实数据库或数仓
- 报表页面或 IM 机器人
- 试点客户的固定汇报模板

这样能避免一开始就陷入“大而全平台建设”，导致试点失焦。

---

## 6. 试点交付范围

建议试点范围尽量小而清晰，控制在一个能被业务方快速理解和验收的边界内。

推荐试点交付范围：

1. 接 1 到 2 份真实预算 / 实际数据源
2. 固定 3 到 5 个高频管理问题
3. 固定 1 条多轮 follow-up 路径
4. 固定 1 套领导汇报说明模板
5. 提供 smoke 脚本和演示文档

试点验收建议看三件事：

- 业务方是否觉得问答速度显著提升
- 分析结果是否足够稳定可信
- 输出说明是否能直接进入日常汇报场景

---

## 7. 试点报价口径

报价不建议按“模型调用次数”去讲，应该按“试点交付价值”去讲。

更适合的口径是：

- 一个预算助手试点包
- 覆盖固定业务范围、固定数据源、固定问题集
- 包含接入、口径梳理、演示、验收、优化

可以按三个层级沟通：

### A. 演示验证版

适合先让客户感受到效果。

范围：

- Demo 环境
- 固定样例数据
- 固定问答路径
- 演示脚本与讲解材料

### B. 业务试点版

适合在一个部门或一类预算主题上落地。

范围：

- 真实数据接入
- 固定问题集
- 基础权限隔离
- 试点验收支持

### C. 扩展落地版

适合在试点证明价值后再谈。

范围：

- 多数据源扩展
- 更多预算分析主题
- 前端或 IM 接入
- 持续运维和优化

这样报价更接近客户对“项目交付”的理解，而不是把自己讲成一个还没包装好的技术实验。
