# PRODUCT_REQUIREMENTS

## 一、背景

企业在日常运营中存在大量信息处理需求：

* 查询制度、流程、规则
* 查询与分析业务数据
* 生成说明、汇报、总结

传统系统：

* 操作复杂
* 依赖人工理解系统结构
* 跨系统协作成本高

本项目目标：

> 通过自然语言，让用户直接驱动企业数据、知识和内容能力。

---

## 二、产品目标

构建一个统一的企业 AI 能力入口，支持：

1. 知识问答
2. 数据分析
3. 内容生成
4. 多轮任务协作（follow-up）

---

## 三、核心用户

* 企业管理人员
* 业务运营人员
* 财务/行政人员
* 未来接入 IM 工具的普通员工

---

## 四、核心场景

### 场景 1：知识问答

用户输入：

> 采购审批要求是什么

系统输出：

* 回答
* 引用来源（citations）

---

### 场景 2：数据分析

用户输入：

> 上个月哪个部门成本最高

系统输出：

* 分析结果
* 表格（artifact）
* SQL（debug）

---

### 场景 3：内容生成

用户输入：

> 根据这个结果写一段汇报

系统输出：

* 正式文本

---

### 场景 4：组合任务

用户输入：

> 查成本最高部门，并写一段说明

系统执行：

* data.analyze
* content.generate

---

### 场景 5：Follow-up

用户输入：

* 再正式一点
* 换成同比
* 补充审批节点

系统应：

* 基于上下文继续执行

---

## 五、功能边界

### 当前必须支持

* 单轮能力调用
* 多步编排
* follow-up continuation
* context 机制
* 统一 response contract

---

### 当前不做

* 复杂 UI 系统
* 权限体系
* 多租户隔离
* 大规模持久化 memory
* 通用 agent 框架

---

## 六、优先级

### P0（当前）

* 编排正确性
* follow-up 能力
* response contract

### P1

* artifact 丰富化
* debug 信息完善

### P2

* workflow
* 多端集成（IM）

---

## 七、验收标准

系统应满足：

1. 能正确识别 intent
2. 能正确路由 capability
3. follow-up 能基于上下文执行
4. response 结构稳定
5. 新能力可以按既有模式扩展

---

## 八、成功标志

* 用户可以只用自然语言完成任务
* 多轮对话不丢上下文
* 数据/知识/内容能力可组合
* API 可被前端或 IM 直接消费
